# SLURM Workloads Reference

Per-script flag matrix, sbatch shape, and copy-pasteable examples for the
launch scripts under `slurm_utils/`. Source of truth: each script's argparse
block (`launch_protein_binder_search.sh:94-148`,
`launch_laproteina_design_pipeline.sh:50-90`,
`launch_laproteina_train.sh:54-90`), plus
`slurm_utils/slurm_helper.sh:484-538` (`generate_*_slurm_header`).

---

## `launch_protein_binder_search.sh`

Four-stage pipeline: **generate → filter → evaluate → analyze**. Each stage
is its own job array; the launcher blocks (`wait_for_job`) between stages.

### Flag matrix

| Flag | Argument | Default | Effect |
|------|----------|---------|--------|
| `--runtime` | `docker` \| `uv` | from `.env` (`CLUSTER_RUNTIME`) | Override the runtime for this invocation |
| `--sweeper` | `FILE` (YAML) | none | Cartesian-product sweep file consumed by `script_utils/generate_inference_configs.py` |
| `--override` | `K=V K=V …` | none | Hydra overrides applied to every generated config; greedy until next `--*` flag |
| `--targets-file` | `FILE` | none | One target per line; runs the pipeline once per target (blank lines and `#` comments ignored) |
| `--num-runs` | `N` | `1` | Number of runs per target in `--targets-file` mode |
| `--on-cluster` | (flag) | off | Skip SSH/rsync — run from the cluster login node |
| `--skip-download` | (flag) | off | Do not rsync results back at the end |
| `--dry-run` / `-n` | (flag) | off | Generate scripts, log all actions, **do not** submit |
| `--help` / `-h` | (flag) | — | Print embedded help |
| `--` | (sep) | — | Stop flag parsing; what follows is positional |

Positional:

| Position | Meaning | Default |
|----------|---------|---------|
| `TARGET_TASK` | Specific target like `22_DerF21` (becomes `++generation.task_name=...`) | none — uses the targets in the pipeline YAML |
| `COUNT` | Run number suffix (only meaningful for tagging) | `1` (silent unless explicitly passed) |

### Stage timing expectations

The script reads `gen_njobs`, `eval_njobs`, `ncpus_`, `run_name` from
`configs/search_binder_pipeline.yaml`. Job array sizes are:

- Generation: `NUM_CONFIGS * gen_njobs` (one array task per config × gen split).
- Filter: `NUM_CONFIGS` (one per config).
- Evaluation: `NUM_CONFIGS * eval_njobs`.
- Analyze: `NUM_CONFIGS`.

`NUM_CONFIGS` is the size of the sweep × targets matrix; with no sweep and one
target it's 1.

### How `run_name` is derived

From `run_single_search` (lines 520–540):

```
{INPUT_RUN_NAME}-search[-{sweeper_basename}][-{TARGET_TASK}][-{COUNT}]
```

- `INPUT_RUN_NAME` = `run_name` field in `configs/search_binder_pipeline.yaml`.
- `sweeper_basename` appended if `--sweeper` is passed.
- `TARGET_TASK` appended if positional or `--override generation.task_name=...`.
- `COUNT` appended **only** when passed explicitly as a second positional.

The remote run directory is `$CLUSTER_ROOT_REMOTE/<run_name>`. `_on-cluster`
mode uses the **current working directory** instead.

### Output layout on the cluster

```
$CLUSTER_ROOT_REMOTE/<run_name>/
├── slurm_batch_script_gen.sh
├── slurm_batch_script_filter.sh
├── slurm_batch_script_eval.sh
├── slurm_batch_script_analyze.sh
├── configs/
│   ├── inference_configs/inf_<i>_<run_name>.yaml
│   └── eval_configs/eval_<i>_<run_name>.yaml
├── inference/                          # generation outputs (PDBs)
├── evaluation_results/                 # evaluation outputs
├── slurm_run_outputs/
│   ├── inf/slurm_<jobid>_<array>.{out,err}
│   ├── filter/slurm_<jobid>_<array>.{out,err}
│   ├── eval/slurm_<jobid>_<array>.{out,err}
│   └── agg/slurm_<jobid>_<array>.{out,err}
├── .env                                # remote .env written by _build_run_env_content
└── .githash
```

### sbatch shape (per stage)

Generated by `generate_array_slurm_header` (`slurm_helper.sh:511-538`):

```
#SBATCH --account     $CLUSTER_ACCOUNT
#SBATCH --time        04:00:00            # default; not overridden by binder script
#SBATCH --job-name    {gen|filter|eval|analyze}_<run_name>
#SBATCH --partition   $CLUSTER_PARTITION
#SBATCH --ntasks-per-node 1
#SBATCH --gpus-per-node   1
#SBATCH --gpus            1
#SBATCH --ntasks          1
#SBATCH --nodes           1
#SBATCH --cpus-per-task   <ncpus_ from YAML>
#SBATCH --open-mode       append
#SBATCH --requeue
#SBATCH --output  ./slurm_run_outputs/<stage>/slurm_%j_%a.out
#SBATCH --error   ./slurm_run_outputs/<stage>/slurm_%j_%a.err
#SBATCH --array=0-<N-1>
```

Where `N` = `NUM_CONFIGS * <split>` (split = `gen_njobs` for generation,
`eval_njobs` for evaluation, `1` for filter / analyze).

### Examples

```bash
# 1. Smallest happy-path: one target, runtime from .env, no sweep
./slurm_utils/launch_protein_binder_search.sh --dry-run 22_DerF21
./slurm_utils/launch_protein_binder_search.sh 22_DerF21

# 2. Force Docker runtime; explicit override-style target
./slurm_utils/launch_protein_binder_search.sh --runtime docker \
    --override generation.task_name=22_DerF21 -- --dry-run

# 3. Sweep beam widths across one target
./slurm_utils/launch_protein_binder_search.sh \
    --sweeper configs/sweeps/beam_width.yaml \
    --override generation.task_name=22_DerF21

# 4. Batch over a targets file, 3 runs per target
./slurm_utils/launch_protein_binder_search.sh \
    --targets-file targets.txt --num-runs 3

# 5. Sitting on the cluster login node already — skip SSH/rsync
./slurm_utils/launch_protein_binder_search.sh --on-cluster 22_DerF21

# 6. Keep results remote (no rsync back)
./slurm_utils/launch_protein_binder_search.sh --skip-download 22_DerF21
```

`targets.txt` format: one target per line; blank lines and lines starting with
`#` are skipped. Example:

```
# Allergens
22_DerF21
23_DerP21

# Cytokines
02_PDL1
```

## `launch_laproteina_design_pipeline.sh`

Three-stage pipeline: **generate → evaluate → analyze**. No filter step. Drives the LaProteina monomer design pipeline (unconditional protein generation, no target / no motif).

### Flag matrix

| Flag | Argument | Default | Effect |
|------|----------|---------|--------|
| `--config-name` / `-c` | `NAME` | `design_monomer_pipeline` | Pipeline config name |
| `--run_name` | `NAME` | from config | Override the derived run name |
| `--runtime` | `docker` \| `uv` | from `.env` | Override runtime |
| `--sweeper` | `FILE` | (none) | YAML sweep file (cartesian product) |
| `--override` | `K=V [K=V …]` | (none) | Config overrides for `generate_inference_configs.py` |
| `--dry-run` / `-n` | (flag) | off | Generate scripts, do not submit |
| `--skip-download` | (flag) | off | Skip rsync of results |
| `--help` / `-h` | (flag) | — | Print embedded help |
| `--` | separator | — | Remaining args treated as Hydra overrides |

Positional:

| Position | Meaning | Default |
|----------|---------|---------|
| `RUN_SUFFIX` | Appended to the derived run name | empty |
| `HYDRA_ARGS...` | Passed verbatim to the inner Hydra invocation | (none) |

The default pipeline config is `configs/design_monomer_pipeline.yaml`. Override with `--config-name <other>` if you have alternative monomer-pipeline configs.

### Output layout on the cluster

```
$CLUSTER_ROOT_REMOTE/<run_name>/
├── slurm_batch_script_{gen,eval,analyze}.sh
├── inference/                        # gen outputs
├── evaluation_results/               # eval outputs
└── slurm_run_outputs/{gen,eval,agg}/slurm_<jobid>_<array>.{out,err}
```

Local download lands at `results_downloaded/<run_name>-<YYYY_MM_DD_HH>/` and
is post-processed by
`python -m proteinfoundation.result_analysis.monomer_merge_results`.

### sbatch shape

Identical to binder search except per-stage time: `generate_array_slurm_header` is called with an explicit `02:00:00` walltime.

### Examples

```bash
# Dry-run first, always
./slurm_utils/launch_laproteina_design_pipeline.sh --dry-run

# Default run
./slurm_utils/launch_laproteina_design_pipeline.sh

# Tag the run with a suffix to differentiate sweeps
./slurm_utils/launch_laproteina_design_pipeline.sh sweep_ckpts

# Switch to UV runtime for one launch
./slurm_utils/launch_laproteina_design_pipeline.sh --runtime uv

# Keep outputs on cluster
./slurm_utils/launch_laproteina_design_pipeline.sh --skip-download
```

## `launch_laproteina_train.sh`

Multi-node distributed training, singleton-requeue chain.

### Flag matrix

| Flag | Argument | Default | Effect |
|------|----------|---------|--------|
| `--runtime` | `docker` \| `uv` | from `.env` | Override runtime |
| `--config` | `FILE` (YAML) | `configs/training_local_latents.yaml` | Training config |
| `--num-jobs` | `N` | `5` | Number of singleton jobs queued (chain length) |
| `--dry-run` / `-n` | (flag) | off | Generate, do not submit |
| `--help` / `-h` | (flag) | — | Print embedded help |
| `--` | (sep) | — | Everything after this is Hydra override |

Hydra overrides can also be passed without `--` — anything not recognized as a
flag is appended to `HYDRA_ARGS` (see lines 85–88).

### Resources from the YAML

The script reads these four keys from the training config
(`launch_laproteina_train.sh:210-213`):

| YAML key | Used as | Default if missing |
|----------|---------|--------------------|
| `nnodes_` | `#SBATCH --nodes` | none — script fails if missing |
| `ngpus_per_node_` | `#SBATCH --gpus-per-node` and `--ntasks-per-node` | same |
| `ncpus_per_task_train_` | `#SBATCH --cpus-per-task` | same |
| `run_name` | run directory name and job name | same |

`yaml_get` searches dot-paths first, then a recursive breadth-first lookup
(`slurm_helper.sh:44-87`), so the keys can be nested anywhere.

### Canonical multi-node example

The 96-GPU binder-finetune size:

```yaml
# in your training YAML
nnodes_: 12
ngpus_per_node_: 8
ncpus_per_task_train_: 8
run_name: laproteina-binder-ft-96gpu
```

Yields:

```
#SBATCH --nodes 12
#SBATCH --gpus-per-node 8
#SBATCH --ntasks-per-node 8
#SBATCH --cpus-per-task 8
```

Total ranks per job: 12 × 8 = 96. PyTorch Lightning's
`SLURMEnvironment(auto_requeue=True)` handles the rank-↔-task mapping.

### Singleton requeue chain (`--num-jobs N`)

`submit_job` is called with `--singleton` (`launch_laproteina_train.sh:255`).
This makes `sbatch` add `--dependency=singleton`, which means **only one job
with this name + user runs at a time**. The launcher submits `N` of them; they
queue up and run sequentially.

Combined with `#SBATCH --requeue` and `#SBATCH --signal SIGUSR1@600` (10 min
before timeout), this gives a "single rolling job" effect: the current one
checkpoints on SIGUSR1, Lightning restarts from checkpoint, and if SLURM
preempts mid-step the job re-queues itself. The `N` jobs in the chain act as
the maximum total wallclock budget = `N * 04:00:00` = 20 h by default.

Resize the chain by passing `--num-jobs`. Single 4-hour run: `--num-jobs 1`.

### sbatch shape

Generated by `generate_train_slurm_header` (`slurm_helper.sh:484-509`):

```
#SBATCH --account     $CLUSTER_ACCOUNT
#SBATCH --time        04:00:00             # hard-coded in launch_laproteina_train.sh:148
#SBATCH --job-name    <run_name>
#SBATCH --partition   $CLUSTER_PARTITION
#SBATCH --ntasks-per-node <gpus_per_node>
#SBATCH --gpus-per-node   <gpus_per_node>
#SBATCH --nodes           <nnodes>
#SBATCH --cpus-per-task   <cpus_per_task>
#SBATCH --open-mode       append
#SBATCH --requeue
#SBATCH --signal          SIGUSR1@600
#SBATCH --output  ./slurm_run_outputs/slurm_%j.out
#SBATCH --error   ./slurm_run_outputs/slurm_%j.err
```

### Output layout

```
$CLUSTER_ROOT_REMOTE/<run_name>/
├── slurm_batch_script_train.sh
├── slurm_run_outputs/slurm_<jobid>.{out,err}
├── checkpoints/                     # if Lightning is configured to write here
├── .env
└── .githash
```

Where checkpoints land depends on `CLUSTER_CKPT_PATH` and your Lightning
trainer's `default_root_dir`. With `CLUSTER_CKPT_PATH` set, Docker mounts it
at `${CLUSTER_CONTAINER_WORKSPACE}/checkpoints` (`slurm_helper.sh:738-744`).

### Examples

```bash
# 1. Dry-run with default config
./slurm_utils/launch_laproteina_train.sh --dry-run

# 2. Custom training config, single 4h job (no requeue chain)
./slurm_utils/launch_laproteina_train.sh --config configs/training_binder_ft.yaml --num-jobs 1

# 3. Docker runtime + Hydra overrides
./slurm_utils/launch_laproteina_train.sh --runtime docker --config configs/training_local_latents.yaml \
    -- log.wandb_project=proteina_binder_ft opt.lr=2e-4
```
